## Context

The session-context bootstrap is produced by one shared renderer (`src/session_context.rs::render_session_context`) feeding four surfaces: the `load_session_context` tool, the `session-context` MCP resource, the `session-context` MCP prompt, and the `GET /v1/context` HTTP endpoint. The default render is ~12 KB. Two sections dominate and are the least scope-specific:

- `<AGENTMEM:TOOLS>` (`{{tools_guide}}`, ~2.8 KB) тАФ generated by concatenating each live tool's description, which duplicates the MCP tool schemas the client already receives. Its only non-schema content (carry the scope keys) is already stated by `{{scope_directive}}`.
- `<AGENTMEM:LAYOUT>` (~3.2 KB) тАФ static vault-mechanics prose that renders identically for every scope, and only matters when the agent writes notes.

SessionStart hooks pull the render over `GET /v1/context`; large outputs are persisted and only the first ~2 KB is reliably injected into the model's context, so the tail (tools guide, layout) is silently lost anyway.

The template engine (`src/template.rs`) is deliberately substitution-only тАФ no conditionals or loops. Anything conditional (which sections appear, "show this only when files are missing") must be resolved in the renderer by choosing a template source or by computing a placeholder value (possibly the empty string).

Template resolution is already layered: per-scope file inside the agents folder тЖТ global env-configured file тЖТ compiled-in default.

## Goals / Non-Goals

**Goals:**
- A small, guaranteed-in-budget **bootstrap** render for session start (scope banner + `PERSONA` + `RULES` + onboarding directive + pointers).
- Keep the **full context** render available on demand, minus the redundant tools guide and minus the relocated layout.
- A dedicated, on-demand **layout** render (resource + HTTP), operator-overridable like the other templates.
- Stop paying for the missing-files onboarding guidance in steady-state sessions.
- Additive HTTP routes and MCP resources; no surface removals.

**Non-Goals:**
- A `verbosity` query parameter on `/v1/context` (rejected in favour of a dedicated `/v1/bootstrap` surface тАФ see Decisions).
- Changing the `session-context` resource/prompt or `load_session_context` tool *surfaces* (their rendered content changes; their shapes do not).
- Adding a bootstrap or layout MCP *prompt*, or a layout *tool* (resources + HTTP only).
- Per-section conditional logic inside the template engine.
- Shrinking operator-authored `PERSONA.md`/`RULES.md` content (a deployment concern, not the server's).

## Decisions

### D1: Dedicated `bootstrap` surface, not a `verbosity` parameter
The lean render is its own surface тАФ `GET /v1/bootstrap` + `agentmem://session-bootstrap/{тАж}` тАФ rather than `GET /v1/context?verbosity=lean`.

- *Why:* `/v1/context` stays byte-for-byte unchanged at the handler level (purely additive routes), and there is no silent behavior change on upgrade тАФ existing hooks pointed at `/v1/context` keep getting the full render; lean is opt-in by repointing the hook. Two named surfaces read more clearly than a mode flag.
- *Alternative considered:* a `verbosity=lean|full` query param on `/v1/context` defaulting to lean. Rejected: it mutates an existing endpoint's default output (silent change for deployed hooks) and threads a mode through a surface that otherwise needs no change.

### D2: Render *kind* selects between two compiled-in default templates
`render_session_context` takes a `kind тИИ { Context, Bootstrap }`. It builds the full context map (all five `{{files.*}}`, `{{scope.*}}`, `{{scope_directive}}`, `{{onboarding_directive}}`) and renders the template resolved for that kind. Each kind has its own layered resolution and its own compiled default:

- `Context`: `AGENT_SESSION_CONTEXT[.scope].md` тЖТ `AGENTMEM_SESSION_CONTEXT_TEMPLATE_FILE` тЖТ `DEFAULT_CONTEXT`.
- `Bootstrap`: `AGENT_SESSION_BOOTSTRAP[.scope].md` тЖТ `AGENTMEM_SESSION_BOOTSTRAP_TEMPLATE_FILE` тЖТ `DEFAULT_BOOTSTRAP`.

`DEFAULT_BOOTSTRAP` references only `{{scope_directive}}`, `{{files.persona}}`, `{{files.rules}}`, `{{onboarding_directive}}`, plus literal prose pointing to the full context (`load_session_context`) and the layout surface. `DEFAULT_CONTEXT` is today's default minus `<AGENTMEM:TOOLS>` and `<AGENTMEM:LAYOUT>`, plus `{{onboarding_directive}}` and a literal one-line layout pointer.

- *Why two templates, not template conditionals:* the engine is intentionally substitution-only. Two sources keeps that property and lets operators override each independently. An operator who only ever uses the hook can author just `AGENT_SESSION_BOOTSTRAP.md`.
- *Alternative considered:* one template with empty-string section values for lean. Rejected: empty `<MEMORY></MEMORY>` tags would still render, and it pushes section markup into code, weakening the override story.

### D3: Drop `{{tools_guide}}` and the `tools` renderer input entirely
Delete the generator and the `<AGENTMEM:TOOLS>` slot; drop the now-dead `tools: &[Tool]` parameter from the renderer and its callers. No back-compat shim тАФ a stray `{{tools_guide}}` in an operator template renders as the literal token (the engine's defined behavior for unknown keys) and is logged once.

- *Why:* it duplicates the MCP tool schemas; its only unique content is already in `{{scope_directive}}`; and it grows the bootstrap every time a tool is added.

### D4: Layout becomes its own scope-parameterized surface
A separate `render_layout(scope)` resolves `AGENT_MEMORY_LAYOUT[.scope].md` тЖТ `AGENTMEM_MEMORY_LAYOUT_TEMPLATE_FILE` тЖТ `DEFAULT_LAYOUT` and renders it through the `Template` engine with the scope context (so overrides may use `{{scope.*}}`; the compiled default has no placeholders). Exposed as `GET /v1/layout` and `agentmem://session-layout/{тАж}`, both scope-required exactly like the context surface.

- *Why scope-required despite scope-independent default content:* uniformity with `/v1/context` and the `session-context` resource, and it enables per-scope layout overrides via the suffix mechanism.
- *Why also an HTTP route, not resource-only:* `load_session_context` is a universally-callable tool, but resources require client resource support тАФ the very gap the HTTP context endpoint exists to bridge. `GET /v1/layout` gives hook/curl clients parity. The bootstrap render keeps a one-line layout pointer so even resource-less, hook-only clients are nudged toward it.
- *Note:* the missing-files onboarding paragraph is **not** part of the static layout тАФ it is session-state-dependent and moves to `{{onboarding_directive}}` (D5).

### D5: `{{onboarding_directive}}` тАФ computed, gated on `missing`
The renderer already computes the `missing` foundational-files list. Set `{{onboarding_directive}}` to the empty string when `missing` is empty, otherwise to the "interview the user, then `evolve_core_persona`" directive (lifted from the old layout prose). Both `DEFAULT_CONTEXT` and `DEFAULT_BOOTSTRAP` reference it тАФ startup happens through the bootstrap hook, so it must appear there. Steady-state sessions render it empty.

- *Naming:* `{{onboarding_directive}}` (not `{{bootstrap_directive}}`) to avoid colliding with the `bootstrap` surface name; it names the situation (onboarding a fresh/partial scope).

## Risks / Trade-offs

- [Lean payload may still exceed the SessionStart inline budget] тЖТ **Measured**: with a representative persona (~0.7 KB) + rules (~1.5 KB) the lean bootstrap renders ~2.5 KB (a fresh/empty scope is ~1.1 KB), down from the old ~12 KB. That edges above the ~2 KB preview threshold observed in one harness. RULES is kept in lean deliberately (safety at turn 1). Mitigation: the lever is operators trimming their own `PERSONA.md`/`RULES.md` тАФ the server cannot shrink authored prose. The drop from 12 KB is the substantive win regardless.
- [Full-render consumers lose inline layout] тЖТ `/v1/context`, `load_session_context`, and the `session-context` resource/prompt no longer embed layout. Mitigation: a one-line pointer in every full and lean render; caps and wrapper-only rules remain server-enforced, so a client that never reads layout degrades to actionable errors, not silent breakage.
- [Operator templates referencing `{{tools_guide}}` break] тЖТ they render the literal token. Mitigation: call it out in the changelog/migration; it is logged once as an unknown placeholder.
- [Terminology overload of "bootstrap"] тЖТ the lean *surface* is `bootstrap` and it contains an *onboarding* directive (formerly "bootstrapping" prose). Mitigation: distinct names (`bootstrap` surface vs `{{onboarding_directive}}`) and a docs sentence.
- [Env-var name drift] тЖТ resolved: new vars use the existing `тАж_TEMPLATE_FILE` suffix (`AGENTMEM_SESSION_BOOTSTRAP_TEMPLATE_FILE`, `AGENTMEM_MEMORY_LAYOUT_TEMPLATE_FILE`) to match `AGENTMEM_SESSION_CONTEXT_TEMPLATE_FILE`.

## Migration Plan

1. Ship the additive routes/resources and the renderer changes together; existing surfaces keep working with shrunken content.
2. Update `docs/session-context-hooks.md` to recommend `GET /v1/bootstrap` as the SessionStart hook target and document `GET /v1/layout` / `agentmem://session-layout`.
3. Changelog: note the `{{tools_guide}}` removal (operator-template breaking) and that full-render content no longer embeds the tools guide or layout.
4. Rollback: revert is clean тАФ the new routes/resources are additive and the renderer change is self-contained; operators with custom templates are unaffected unless they used `{{tools_guide}}`.
